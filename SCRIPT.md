# SCRIPT.md - Quality Engineering vs. Quality Assurance Slide Text

This document lists the exact text copy that will appear on screen for the video. Since the video has no voiceover narration, these text lines carry the full educational message.

---

## Beat 1: Title Card
- **Header:** Quality Engineering
- **Subtitle:** Evolving from finding bugs to building quality in.
- **Presenter:** Andrew Darnell, Principal Test Engineer

## Beat 2: Traditional QA
- **Header:** Traditional Quality Assurance (QA)
- **Concept:** Reactive testing at the end of the development cycle.
- **Key Points:**
  - Focuses on discovering defects after they are introduced.
  - Separate phase by a siloed team.
  - Heavy reliance on manual repetition.
- **Quote:** "We can't release yet, QA is still testing."

## Beat 3: Quality Engineering
- **Header:** Quality Engineering (QE)
- **Concept:** Proactive integration of quality throughout the lifecycle.
- **Key Points:**
  - Same QA skills turned into continuous assurance.
  - Focuses on preventing defects and improving processes.
  - Integrated directly into CI/CD pipelines.
- **Quote:** "Quality is built into the pipeline, not audited at the end."

## Beat 4: Key Differences
- **From Traditional QA:**
  - Reactive finding of bugs
  - Manual Test Execution
  - Siloed gatekeepers
  - A bottleneck to release
- **To Quality Engineering:**
  - Proactive mitigation of risk
  - Test Automation & Infrastructure
  - Embedded Quality Experts
  - An accelerator for speed and scale

## Beat 5: Feedback Loops
- **Header:** Feedback Loops
- **Key Points:**
  - Continuous feedback via unit and integration tests.
  - Shifting left: working with Devs to enable early checks.
  - Loop: Code -> Build -> Auto-Test -> Feedback.

## Beat 6: Sprint Simulation
- **Header:** The Scaling Problem vs. The Automation Engine
- **Counters:** Built, Verified, Escaped Bugs.
- **Scenario Left:** QA Bottleneck (Manual capacity hits a wall)
- **Scenario Right:** QE Automation (Scalable capacity grows with code)

## Beat 7: ROI
- **Header:** Debunking the "Too Expensive" Fallacy
- **Concept:** Zero Risk Appetite.
- **Key Points:**
  - Typical projects take 6-36 months and run for years.
  - Upfront automation effort pays back over 72-120 releases.
  - Exponential long-term savings.
